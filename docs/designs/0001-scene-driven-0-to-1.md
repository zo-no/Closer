# 场景驱动的高质量认识与 0-1 推进 Design

## Requirement Reference
- docs/requirements/0001-scene-driven-0-to-1.md

## Design Goal

第一版设计的目标不是打造一个全能恋爱助手，而是围绕用户最真实的前置卡点，设计一套更轻、更有方法感的产品形态：

- 帮用户找到更适合自己的认识场景
- 帮用户判断哪些场景值得投入
- 帮用户在进入场景前后获得轻量指导
- 帮用户把“浪漫遇见”与“实际行动”结合起来

第一版重点支持的场景先聚焦在两类：

1. 读书 / 分享 / 小型文化活动
2. 摄影 / 拍照 / 视觉表达场景

这两类场景更接近目标用户的真实审美偏好，也更容易产生自然、文艺、带有故事感的相遇。

## User Flow

第一版用户流程建议设计为：

1. 用户先表达自己的偏好画像
   - 喜欢什么类型的人
   - 对什么气质和互动方式有感觉
   - 哪些场景自己天然更愿意进入

2. 产品基于画像提供“优先场景方向”
   - 不直接给所有活动堆列表
   - 而是给出更适合这个用户的场景建议

3. 用户选择一个本周可执行的场景方向
   - 例如一场读书活动或一次摄影相关活动

4. 产品为该场景提供轻量行动支持
   - 为什么这个场景适合你
   - 到场前的准备建议
   - 在现场如何自然开启互动
   - 哪些信号说明值得继续推进

5. 场景结束后做一次简短复盘
   - 有没有遇到合适的人
   - 哪个环节卡住了
   - 这类场景是否值得继续投入

## Functional Shape

第一版功能形态建议由 4 个部分构成：

### 1. 场景画像模块

作用：
- 将“我适合去哪里认识人”具体化

输入：
- 用户偏好类型
- 生活节奏
- 时间预算
- 喜欢的互动气质

输出：
- 更适合当前用户的场景方向排序

### 2. 场景建议模块

作用：
- 给出当前阶段最值得尝试的场景类型，而不是泛泛推荐

重点不是做大而全活动信息平台，而是先做：
- 场景类别判断
- 场景适配原因解释
- 进入门槛提示

### 3. 场景内推进模块

作用：
- 帮助用户从“到了现场但不知道怎么自然开始”进入“有轻量行动支撑”

这里不做强操控型话术，而是提供：
- 自然互动切入点
- 不同场景下的低压力开场方式
- 初步判断对方是否值得继续认识的提示

### 4. 复盘模块

作用：
- 让用户不只是去一次活动，而是逐渐形成自己的有效场景地图

输出：
- 这个场景对自己是否有效
- 自己在哪一段卡住了
- 这个方向要继续、调整还是放弃

## Key Interactions

第一版设计里，关键交互不是复杂 UI，而是以下几种判断节点：

1. 用户先完成“偏好和场景画像”
2. 产品给出场景方向建议，而不是直接塞大量候选列表
3. 用户选定本周要尝试的场景
4. 产品给出该场景下的轻量行动提示
5. 用户结束后完成一次复盘

也就是说，产品更像“场景驱动的行动陪跑器”，而不是“机会池管理工具”。

## Tradeoffs

### 保留的方向

- 强调真实场景
- 强调浪漫感和自然感
- 强调用户与场景的匹配，而不是单纯追求认识人数
- 将方法论、引导和记录结合进产品

### 暂不优先的方向

- 不先做大规模活动聚合平台
- 不先做复杂匹配算法
- 不先做深度聊天生成器
- 不先做见面后的完整关系管理

### 取舍原因

如果第一版过早做成“平台 + 内容库 + CRM + AI 助手”的混合体，范围会失控，也会继续模糊最真实的问题。第一版必须先把“找到对的场景，并在其中自然推进”这件事做透。

## Open Questions

1. 第一版场景建议是否需要真实活动数据源，还是先从“场景类别建议 + 手动输入”开始
2. 摄影场景是否应该作为重点方向，还是作为后续补充方向
3. 用户在现场需要的是简短提示卡，还是完整的方法教程
4. 复盘结果是否需要反向更新用户画像和场景排序

## Handoff To Spec

下一步 spec 应该重点回答：

1. 第一版 MVP 到底做成一个什么样的界面和交互入口
2. 第一版是否需要真实活动列表，还是先用场景类型和示例活动验证
3. 偏好画像、场景建议、行动提示、复盘这 4 个模块的最小实现边界是什么
4. 第一版测试要如何验证：
   - 用户是否更愿意进入目标场景
   - 用户是否更容易在场景中发起自然互动
   - 用户是否能更清楚地判断某类场景对自己是否有效
